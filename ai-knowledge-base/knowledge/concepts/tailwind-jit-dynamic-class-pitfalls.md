# Tailwind JIT Dynamic Class Pitfalls

## Overview
Tailwind CSS JIT (Just-In-Time) mode scans source files at build time and generates only the CSS classes it finds as static strings. Several patterns in the Beldify admin codebase produce classes that look valid in development but fail silently in production because JIT never sees them. These are correctness bugs, not style issues.

## Key Points
- **Phantom tokens** (`tw-primary-*`): custom color names not in `tailwind.config.js` render as invisible/unstyled elements — no visual error, just missing background/text color
- **Dynamic class construction** (`tw-bg-{{ $var }}-100`): JIT cannot see runtime values — the class string is never in source, never generated, never delivered to the browser in production CSS
- **`@apply` in Blade inline `<style>`**: PostCSS only processes source files through Vite's build pipeline; `@apply` directives embedded in Blade templates are emitted as raw text to the browser and silently ignored
- **Mixed BS4/BS5 modal syntax**: `data-toggle`/`data-dismiss` (Bootstrap 4) co-existing with `data-bs-toggle`/`data-bs-dismiss` (Bootstrap 5) — only the BS5 syntax functions when the BS5 bundle is loaded
- **Prefix + pseudo-class ordering**: `tw-hover:bg-gray-50` is invalid — must be `hover:tw-bg-gray-50` when using Tailwind's `prefix` config option

## Details

### Phantom token example
In the Beldify commissions cluster, 64 occurrences of `tw-bg-primary-600`, `tw-text-primary-600`, `hover:tw-bg-primary-700`, `focus:tw-ring-primary-500` appear across `commissions/index.blade.php` and related files. The `primary` color scale is never defined in `tailwind.config.js`. Tailwind generates nothing for these classes; the buttons and links that use them render with no background color or focus ring.

**Fix**: replace `tw-primary-*` with real Atlas tokens — `tw-bg-indigo-700`, `tw-text-indigo-700`, `hover:tw-bg-indigo-800`, `focus:tw-ring-amber-300`.

### Dynamic class construction example
```blade
{{-- This class is never generated by JIT --}}
<span class="tw-bg-{{ $commission->getStatusColor() }}-100 tw-text-{{ $commission->getStatusColor() }}-800">
```
`getStatusColor()` returns `'yellow'`/`'green'`/`'red'`/`'blue'` at runtime. JIT scans the source and sees only the template literal — it does not evaluate PHP. The generated CSS bundle never includes `tw-bg-yellow-100` etc. from this file. In development (without JIT purging), all Tailwind classes are available so the bug is invisible.

**Fix**: use a static `match` or `@switch` Blade conditional that lists every class string literally:
```blade
@php
    $badgeClass = match($commission->status) {
        'approved' => 'tw-bg-amber-100 tw-text-amber-800',
        'rejected' => 'tw-bg-rose-100 tw-text-rose-800',
        default    => 'tw-bg-indigo-100 tw-text-indigo-800',
    };
@endphp
<span class="{{ $badgeClass }}">
```
Or add the dynamic class combinations to Tailwind's `safelist` in `tailwind.config.js`.

### `@apply` in Blade inline style
```blade
{{-- banners/index.blade.php lines 8-52 --}}
<style>
    .banner-card { @apply tw-rounded-lg tw-transition-all; }
</style>
```
PostCSS `@apply` directives are only resolved when the CSS file passes through Vite's build pipeline (e.g., in `.css` or `.scss` source files). A `<style>` block inside a `.blade.php` template is served as raw HTML — `@apply` is emitted literally to the browser, where it is not a valid CSS declaration and is ignored. The resulting element has no `border-radius` or transition.

**Fix**: move the styles to a proper `.css` source file that Vite processes, or convert to inline Tailwind utility classes directly on the element.

### CI lint guard (`.github/workflows/admin-lint.yml`)
After session 1295f6ce applied all four fixes, a CI workflow was added to prevent regression on every PR touching admin Blade files:

```yaml
# .github/workflows/admin-lint.yml
name: Admin Blade Lint
on: [push, pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: No dynamic class construction
        run: |
          ! grep -rn 'class=".*tw-[a-z]+-{{ ' resources/views/admin/
      - name: No @apply in Blade templates
        run: |
          ! grep -rn '@apply' resources/views/admin/
      - name: No mixed BS4/BS5 modal syntax
        run: |
          ! grep -rn 'data-toggle=' resources/views/admin/
      - name: No tw-primary-* without safelist
        run: |
          ! grep -rn 'tw-[a-z]*-primary-' resources/views/admin/
```

Each rule is a negative grep — the job fails if the pattern is found. This guards against the four known anti-patterns being re-introduced.

Additionally, `tw-ring-primary-*` tokens (used in focus rings via JS components, not Blade) were added to the Tailwind `safelist` in `tailwind.config.js`:

```js
// tailwind.config.js
module.exports = {
  safelist: [
    { pattern: /^tw-ring-(primary|indigo|amber)-\d+$/ },
  ],
  // ...
}
```

This is the correct escape hatch for classes that legitimately cannot be static strings (e.g., classes injected by third-party JS). The CI rule only bans `tw-primary-*` *in Blade templates*; safelist is the right solution for runtime JS injection.

### Admin CSS bundle context (discovered 2026-05-29)
A fifth variant specific to the admin surface: admin pages load `admin/app-assets/app.css` (the legacy PixInvent/Bootstrap stylesheet compiled separately) **not** the Vite-built `build/assets/app.css`. Tailwind utility classes like `tw-bg-emerald-600`, `tw-text-white`, `tw-rounded-lg` are only present in the Vite build output.

**Symptom**: Quick Actions buttons on admin community pages rendered with no background color or shape despite correct Tailwind class names in the template — the classes simply do not exist in the CSS file the page loads.

**Fix**: Inline `<style>` block with semantic class names scoped to the admin component:
```blade
<style>
.qa-btn         { display: flex; align-items: center; gap: .5rem; padding: .5rem .75rem; border-radius: .375rem; font-size: .875rem; font-weight: 500; border: none; cursor: pointer; width: 100%; }
.qa-btn-success { background: #059669; color: #fff; }
.qa-btn-warning { background: #d97706; color: #fff; }
.qa-btn-danger  { background: #dc2626; color: #fff; }
.qa-btn-info    { background: #4f46e5; color: #fff; }
</style>
```

This pattern avoids Tailwind entirely for admin-specific components where the Vite bundle is not loaded, while still producing Atlas-consistent colors.
