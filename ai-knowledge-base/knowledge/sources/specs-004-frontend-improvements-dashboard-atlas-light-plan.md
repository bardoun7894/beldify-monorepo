## Critical Pre-Work: How `tw-` Colors Actually Work

### Finding
`public/css/tailwind.css` is a **committed, pre-compiled 108 KB minified file** — not generated by Vite (`vite build` writes to `public/build/`, not `public/css/`). The seller shell loads it via `<link rel="stylesheet" href="{{ asset('css/tailwind.css') }}">` with a static `file_exists()` guard.

Tailwind compiles all color classes as hardcoded `rgb()` values (`tw-bg-indigo-50 { background-color: rgb(238 242 255 / ...) }`). CSS custom property injection cannot remap these at runtime.

### Consequence for the re-theme
The v3 components (`badge`, `btn`, `kpi-tile`) use Tailwind color tokens (`tw-bg-indigo-50`, `tw-text-amber-700`, etc.) **and are shared with 25 admin pages and 78 non-dashboard seller pages**. Editing the components themselves would re-theme the entire admin/seller section — beyond this task's scope.

### Chosen mechanism — dashboard-scoped CSS override block
All Atlas color corrections for the dashboard are delivered via a scoped `<style>` block in `dashboard.blade.php`'s existing `@section('styles')`, targeting:
- `.sdash-rail` and its descendants (light layer)  
- `.data-dash-section[data-dash-section="complex"]` and its descendants (heavy layer)  

These selectors are already unique to the dashboard view. No component files are touched. The Tailwind CSS rebuild step is **not required** for any of the four files listed.
